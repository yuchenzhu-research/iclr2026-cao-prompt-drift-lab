# 04_results_analysis

This file is a provenance note for the numbers reported in the paper. It specifies what is aggregated, what is excluded, what files are the source of truth, and how any row can be traced back to preserved raw evidence.

Non-goals:
- Not an evaluation-rule document (rules live under `reproducibility/03_evaluation_rules/`).
- Not a plotting/figure document (figures are generated by scripts under `reproducibility/tools/figures/`).
- No record rewriting, relabeling, reweighting, or manual edits.

---

## Directory contract

`04_results/` is a one-way pipeline:

1. `01_raw_model_outputs/` — preserved generator outputs (PDF)
2. `02_raw_judge_evaluations/` — preserved judge bundles (JSON)
3. `03_processed_evaluations/` — canonical processed records and summary tables

**Numeric authority**
- Paper-citable numbers come **only** from:
  - `reproducibility/04_results/03_processed_evaluations/**/summary_tables/*.csv`

---

## Inputs used for aggregation

Canonical processed records (record-level JSON):
- `reproducibility/04_results/03_processed_evaluations/<judge_version>/valid_evaluations/**/*.json`

Preserved upstream evidence:
- `reproducibility/04_results/02_raw_judge_evaluations/`
- `reproducibility/04_results/01_raw_model_outputs/`

---

## Record inclusion and exclusion

Inclusion is determined by validity criteria defined under:
- `reproducibility/03_evaluation_rules/`

If exclusions exist, they are recorded at:
- `reproducibility/04_results/03_processed_evaluations/<judge_version>/summary_tables/excluded_records.jsonl`

If `excluded_records.jsonl` is absent, interpret it as **0 exclusions** for that judge version.

---

## Grouping keys and the no-merge rule

No-merge rule: results from different `judge_version` values are never pooled.

Aggregations are computed using explicit fields present in `scores_long.csv`.
Common fields include:
- `generator_model`
- `question_id`
- `judge_version`

`prompt_variant` and `trigger_type` are part of the canonical slice keys for all shipped judge versions.

---

## Canonical summary tables

For each `<judge_version>`, canonical outputs are written to:
- `reproducibility/04_results/03_processed_evaluations/<judge_version>/summary_tables/`

Contents:
- `scores_long.csv` — one row per evaluated file (valid records only)
- `scores_grouped.csv` — grouped means over valid rows
- `run_meta.json` — run metadata for auditing
- `excluded_records.jsonl` — excluded rows with reasons (optional; only when exclusions are non-empty)

---

## Deterministic score composition

`total` is computed deterministically as:

```
total = A_structure
      + B_snapshot_constraint
      + C_actionability
      + D_completeness
      + E_drift_failure
```

No reweighting, smoothing, normalization, or manual adjustment is applied.

---

## Traceability (row → record → raw)

### Row → processed record JSON

Each row in `scores_long.csv` corresponds to exactly one processed record JSON stored under:
- `reproducibility/04_results/03_processed_evaluations/<judge_version>/valid_evaluations/**/*.json`

**Do not assume any record filename pattern** (filenames may be hash-based). Locate records by matching JSON fields.

Recommended match strategy:

- **Primary key (preferred when present):** `record.file`
  - Canonical rebuilds also populate `record.raw_pdf`, which directly identifies the preserved PDF under `01_raw_model_outputs/`.

- **Fallback keys:** fields that are present in both `scores_long.csv` and the record JSON, typically:
  - `generator_model`
  - `question_id`
  - (optionally) `judge_model` / bundle metadata fields
  - `prompt_variant` / `trigger_type`

### Processed record JSON → raw PDF

Raw PDFs are stored under:
- `reproducibility/04_results/01_raw_model_outputs/`

`record.file` semantics:
- `record.file` preserves the original bundle-side file label.
- Canonical rebuilds additionally store `record.raw_pdf` as the normalized repo-relative PDF path.

**Naming conventions**
- Where a naming template is used (e.g., `q{question_id}_...pdf`), treat it as a convenience for navigation, not as a hard contract.
- If a template lookup fails, use directory search within `01_raw_model_outputs/` for the matching question/model context.
