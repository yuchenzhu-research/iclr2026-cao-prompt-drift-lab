# 04_results_analysis

This file is a provenance note for the **numbers reported in the paper**. It specifies (i) what is aggregated, (ii) what is excluded, (iii) what files are the source of truth, and (iv) how any row can be traced back to preserved raw evidence.

**Non-goals**
- Not an evaluation-rule document (rules live under `supplement/03_evaluation_rules/`).
- Not a plotting/figure document (figures are generated by scripts under `supplement/tools/figures/`).
- No record rewriting, relabeling, reweighting, or manual edits.

---

## Directory contract

`04_results/` is organized as a one-way pipeline:

1. `01_raw_model_outputs/` — preserved generator outputs (PDF)
2. `02_raw_judge_evaluations/` — preserved judge bundles (JSON)
3. `03_processed_evaluations/` — canonical per-file records and summary tables

Only `03_processed_evaluations/**/summary_tables/` is used as the **paper-citable numeric source**.

---

## Inputs used for aggregation

### Canonical per-file records

Aggregation consumes processed records only:

- `supplement/04_results/03_processed_evaluations/<judge_version>/valid_evaluations/**/record_*.json`

### Preserved upstream evidence

- `supplement/04_results/02_raw_judge_evaluations/`
- `supplement/04_results/01_raw_model_outputs/`

---

## Record inclusion and exclusion

Inclusion is determined by validity criteria defined under:

- `supplement/03_evaluation_rules/`

Records marked invalid are excluded from all numeric summaries and listed at:

- `supplement/04_results/03_processed_evaluations/<judge_version>/summary_tables/excluded_records.jsonl`

Validated records are treated as immutable aggregation inputs.

---

## Grouping keys and the no-merge rule

All aggregations are computed using explicit fields present in `scores_long.csv`:

- `generator_model`
- `prompt_variant`
- `question_id`
- `trigger_type`
- `judge_version`

**No-merge rule:** results from different `judge_version` values are never pooled.

---

## Canonical summary tables

For each `<judge_version>`, canonical outputs are written to:

- `supplement/04_results/03_processed_evaluations/<judge_version>/summary_tables/`

Contents:

- `scores_long.csv` — one row per evaluated file (valid records only)
- `scores_grouped.csv` — grouped means over valid rows
- `excluded_records.jsonl` — excluded rows with reasons
- `run_meta.json` — run metadata for auditing

### Deterministic score composition

`total` is computed deterministically as:

```
total = A_structure
      + B_snapshot_constraint
      + C_actionability
      + D_completeness
      + E_drift_failure
```

No reweighting, smoothing, normalization, or manual adjustment is applied.

---

## Traceability (row → record → raw)

Each row in `scores_long.csv` corresponds to exactly one processed record:

- `supplement/04_results/03_processed_evaluations/<judge_version>/valid_evaluations/**/record_*.json`

### Join keys (canonical)

- `generator_model`
- `question_id`
- `prompt_variant`
- `trigger_type`

### Raw PDF resolution rule

Raw PDFs are stored under:

- `supplement/04_results/01_raw_model_outputs/<generator_model_dir>/`

and follow the naming template:

- `q{question_id}_{prompt_variant}_{trigger_type}.pdf`

### `record.file` field semantics

- `v0`: `record.file` may be an identifier from the judge bundle and is not required to be a repo path.
- `v1/v2`: `record.file` is a repo-relative pointer under `01_raw_model_outputs/`.

If `record.file` is not repo-relative (v0), locate the raw PDF using the join keys and the naming template.

---

## Failure taxonomy

Two layers are kept separate:

- Rubric scores (A–E) used for numeric summaries.
- Protocol-level failure flags used to exclude unusable records.

Failure flag definitions live at:

- `supplement/03_evaluation_rules/failure_taxonomy.md`

Narrative labels used in the paper are descriptive only and do not change protocol semantics.
